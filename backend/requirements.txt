# Model training + inference
# Plain `tensorflow` on every platform, including Apple Silicon — since
# TF 2.13, the standard PyPI package ships native macOS arm64 wheels, so the
# separate `tensorflow-macos` package is no longer needed. Mixing
# `tensorflow-macos` with `tensorflow-metal` is a common source of exactly
# the crash this used to cause here: dlopen("libmetal_plugin.dylib")
# failing with "Library not loaded: @rpath/_pywrap_tensorflow_internal.so"
# because the two packages' internal binaries drift out of sync with each
# other. `tensorflow-metal` alone (GPU/Neural Engine acceleration, optional)
# is fine paired with plain `tensorflow` — pip resolves a compatible pair
# together in one pass instead of two separately-versioned packages.
tensorflow
tensorflow-metal; sys_platform == "darwin" and platform_machine == "arm64"
tf-keras
